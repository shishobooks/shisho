# Cache Management — Design

**Date:** 2026-04-23
**Status:** Draft

## Goal

Give admins a place in the Settings UI to see how much disk space each server cache is using and clear any of them on demand.

## Context

Shisho maintains three on-disk caches, all rooted at `config.CacheDir` (default `/config/cache`):

| Cache | Location | Package | Purpose |
|-------|----------|---------|---------|
| Downloads | `{cache_dir}/downloads` and `{cache_dir}/downloads/bulk` | `pkg/downloadcache` | Generated format conversions (e.g. kepub), plugin-generated files, bulk zips |
| CBZ pages | `{cache_dir}/cbz/{fileID}/page_*.{ext}` | `pkg/cbzpages` | Page images extracted from CBZ files for the reader viewer |
| PDF pages | `{cache_dir}/pdf/{fileID}/page_*.jpg` | `pkg/pdfpages` | Rendered JPEGs for PDF pages in the reader viewer |

`downloadcache` already has LRU-based auto-cleanup against `download_cache_max_size_gb`; `cbzpages` and `pdfpages` have no size cap. There is no UI today for observing or clearing any of these.

The existing admin area lives under `/settings/*` with a sidebar driven by `useAdminNavItems()`. `/settings/server` (`AdminSettings.tsx`) shows read-only config.

## Non-goals

- Auto-cleanup for page caches (could be added later if needed).
- Per-file or per-book cache inspection.
- Cached size reporting with background invalidation, SSE streaming, or server-side caching of size results.
- Exposing the raw file list on disk.

## User-facing behavior

A new `Cache` item appears in the Settings sidebar (after `Jobs`, before `Logs`, using a `HardDrive` or `Database` lucide icon). Visible to any role with `config:read` — matches how `/settings/server` is gated today.

The page shows three cards, one per cache, each with:

- Cache name and a one-line description.
- Total size in human-readable form (e.g., `124 MB`).
- File count.
- A `Clear` button.

`Clear` is disabled for users without `config:write`, and while a clear is in flight. Clicking it opens an `AlertDialog` (using the existing Radix-based dialog pattern) with copy like:

> Clear {cache name}? This will delete {N} files ({size}). Content will be regenerated on next access.

Confirming triggers the clear request, toasts the result on success/failure, and refetches the cache list so the card shows the new (typically zero) size.

## Architecture

### Backend

Per-cache methods are added where each cache type lives, keeping on-disk layout knowledge encapsulated.

On each of `downloadcache.Cache`, `cbzpages.Cache`, `pdfpages.Cache`:

```go
// SizeBytes returns total size in bytes and file count of the cache.
// Missing root directories are treated as empty (0, 0, nil).
func (c *Cache) SizeBytes() (bytes int64, count int, err error)

// Clear removes all cached content but preserves the root directory itself.
// Safe to call when the directory does not exist (treated as no-op).
func (c *Cache) Clear() error
```

- `downloadcache` reuses existing `GetTotalCacheSize` plus bulk zip listing to compute size; `Clear` removes children of the root dir (including the `bulk` subdirectory) but not the root itself.
- `cbzpages` and `pdfpages` walk `{cache_dir}/cbz` and `{cache_dir}/pdf` respectively.

A new `pkg/cache/` package owns HTTP exposure:

- `pkg/cache/handlers.go` — `Handler` struct with the three cache instances injected.
- `pkg/cache/routes.go` — registers routes on the authenticated API group.

Endpoints:

| Method | Path | Permission | Response |
|--------|------|------------|----------|
| `GET` | `/api/cache` | `config:read` | `{ caches: [{ id, name, description, size_bytes, file_count }] }` |
| `POST` | `/api/cache/:id/clear` | `config:write` | `{ cleared_bytes, cleared_files }` |

`:id` ∈ `{"downloads", "cbz_pages", "pdf_pages"}`. Unknown `:id` returns 404. `id` is the stable machine identifier; `name` is the human-readable label.

Dispatch is a simple `switch` on `:id` inside the handler. No interface/registry abstraction — three caches does not justify it, and each cache already owns its own logic.

Errors from `Clear()` are logged via the existing logger and returned as 500 with a safe message.

### Frontend

- `app/router.tsx` — register `/settings/cache` under the existing settings layout, wrapped in a `ProtectedRoute` requiring `config:read`.
- `app/components/pages/useAdminNavItems.ts` — add a `Cache` entry gated on `canViewConfig`.
- `app/components/pages/AdminCache.tsx` — new page rendering the three cards and the confirm dialog.
- `app/hooks/queries/cache.ts` — `useCaches()` (React Query `useQuery` on `GET /api/cache`) and `useClearCache()` (`useMutation` on `POST /api/cache/:id/clear`, invalidates the `['cache']` query on success).

TypeScript types for request/response structs are generated by `mise tygo` from the Go source; no hand-written types.

The page uses existing primitives (`Button`, `AlertDialog`, toast) and the established snake_case JSON payload convention.

## Error handling

- `SizeBytes()` returns `(0, 0, nil)` for a missing root directory; real IO errors propagate and surface as a per-card error state in the UI ("Failed to compute size — retry?").
- `Clear()` is idempotent when the directory is already empty or missing.
- Permission denials (e.g., user without `config:write` calling clear directly) return 403 from the existing permission middleware.

## Testing

**Backend (unit, with `t.Parallel()`):**

- `pkg/downloadcache/cache_test.go` — add tests for `SizeBytes` (empty, populated including bulk zips, file count correct) and `Clear` (removes children, preserves root, idempotent).
- `pkg/cbzpages/cache_test.go` — same coverage for CBZ page cache.
- `pkg/pdfpages/cache_test.go` — same coverage for PDF page cache.
- `pkg/cache/handlers_test.go` — table-driven: list returns all three with correct sizes; clear by id dispatches to the right cache; unknown id returns 404; permissions are enforced by middleware (asserted via route-registration test or integration test depending on existing patterns in the codebase).

**Manual verification:**

- Start dev server, visit `/settings/cache`, confirm three cards render with sizes.
- Clear each cache and confirm the card shows zero and the files on disk are gone but the root directories remain.
- Verify a Viewer role sees sizes but no Clear button; an Editor likewise (since `config:write` is admin-only today).

No E2E coverage for the first cut; the feature is low-risk and backend tests cover the core behavior.
