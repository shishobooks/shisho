# M4B Audiobook Tagging ‚Äî Design

## Problem

When an M4B file generated by Shisho is opened in audiobook players like Bound (iOS), the title displays as "Unknown" even though the file has a valid `¬©nam` title atom. Reproducer: `/Users/robinjoseph/Downloads/[Caro Claire Burke] Yesteryear {Rebecca Lowman}.m4b` ‚Äî our parser reads title "Yesteryear" correctly, but Bound shows "Unknown".

Root cause: most audiobook players follow the convention **Title (`¬©nam`) = chapter, Album (`¬©alb`) = book title**. Shisho currently writes `¬©alb` only when the book has a series (formatted as `"Series Name #N"`). Books with no series get an empty `¬©alb`, so Bound falls back to "Unknown".

Additionally:
- `stik` (media type) is only written when the source file had one, so Shisho-generated files often have no audiobook marker.
- The book description is written to `desc` but not to `¬©cmt`, which some players and taggers prefer.

## Goals

1. Book title reliably displays in audiobook players (Bound, Overcast, Audible, Plex, Audiobookshelf).
2. Series info remains preserved and readable through a round-trip (write ‚Üí re-scan).
3. Description appears in both `desc` and `¬©cmt`.
4. Generated files are unambiguously tagged as audiobooks.

## Design

### Writer changes (`pkg/mp4/writer.go`, `pkg/filegen/m4b.go`)

**`¬©alb` (Album) ‚Äî always the book title.** Drop the "Series Name #N" album formatting. Album now mirrors the title regardless of whether the book has a series. This is the standard audiobook convention and the primary fix for Bound's "Unknown" display.

**`¬©grp` (Grouping) ‚Äî series in `"Series Name #N"` format.** When the book has a series, write it here. If no series, omit the atom.

**Freeform Audible-style series atoms.** When the book has a series, write:
- `----:com.apple.iTunes:SERIES` = series name
- `----:com.apple.iTunes:SERIES-PART` = series number (as string; `"1"`, `"1.5"`)

This is what Audible natively uses and what Tone writes. Required for full compatibility with modern audiobook managers.

**`stik` (Media Type) ‚Äî always `2` (audiobook).** The M4B generator produces audiobooks by definition, so hardcode `MediaType = 2` in `buildMetadata` instead of preserving `src.MediaType`. The writer branch that gates on `MediaType > 0` stays the same; only the value fed to it changes.

**`¬©cmt` (Comment) ‚Äî mirror book description.** When `book.Description` is set, write it to both `Metadata.Description` (which becomes `desc`) and `Metadata.Comment` (which becomes `¬©cmt`).

### Reader changes (`pkg/mp4/reader.go`, `pkg/mp4/metadata.go`)

**Series parsing priority:**
1. `com.apple.iTunes:SERIES` + `com.apple.iTunes:SERIES-PART` freeform atoms (new)
2. `¬©grp` Grouping (existing behavior)

**Drop `¬©alb` as a series source.** Since the writer no longer puts series info in Album, the reader should not try to parse series out of it either. Files generated by the new writer will have `¬©alb` = book title, and parsing series from book title would produce garbage. Pre-existing Shisho-generated files that stored series in Album will lose their series info on re-scan ‚Äî the user has accepted this tradeoff (series data is still in the database; only the embedded file tag changes).

Album continues to be read into `Metadata.Album` for informational purposes, but is not a series source.

### Metadata struct

No new fields needed. The existing `Metadata.Series` and `Metadata.SeriesNumber` are populated by the new parse priority, and the existing `Metadata.Freeform` map already round-trips freeform atoms ‚Äî the writer just needs explicit branches for `com.apple.iTunes:SERIES` / `SERIES-PART` so they aren't double-written by the generic `Freeform` loop. Add these keys to the `explicitFreeformKeys` skip set in `buildIlst`.

## Out of scope

- Other audiobook-specific atoms (e.g. `¬©wrk` Work, `pcst` Podcast flag, chapter-level images).
- Plex-specific or Audiobookshelf-specific custom atoms beyond what's already covered.
- Changes to EPUB or CBZ metadata writing.
- Migrating existing in-library M4B files in place ‚Äî users re-download through the existing generator path to pick up the new tags.

## Testing

- `pkg/mp4/reader_test.go` ‚Äî series read from `com.apple.iTunes:SERIES` / `SERIES-PART` freeform atoms; series read from `¬©grp` as fallback; Album is NOT parsed as series.
- `pkg/mp4/writer_test.go` / `mp4_test.go` ‚Äî round-trip: book with series writes freeform + grouping; book without series has empty grouping and no series freeform; Album always equals title; `stik` always `2`; `¬©cmt` written alongside `desc`.
- `pkg/filegen/m4b_test.go` ‚Äî end-to-end: generating an M4B with series produces the expected atoms; generating without series writes title to Album and no series atoms; description flows to both `desc` and `¬©cmt`.
